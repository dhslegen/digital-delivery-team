### B3. `aggregate.mjs` / `report.mjs` 未关闭 SQLite 连接

**位置**：`bin/aggregate.mjs:21-22` / `bin/report.mjs:72-73` / `bin/baseline.mjs`（同样模式）

**现象**：

```js
const store = new DeliveryStore(DB);
await store.openOrCreate();
// ... 业务逻辑 ...
process.exit(0);   // ← 没有 store.close()
```

`bin/lib/store.mjs:22-26` 提供了 `close()` 方法但全栈无人调用。

**影响**：Node 进程退出时 v8 会回收，单次执行下不会泄漏。但：
1. 若被嵌入到长生命周期进程（如未来可能的 watch 模式 / MCP server），SQLite 句柄会累计；
2. 异常路径（比如 schema.sql 读取失败抛出）下，DB 句柄持有锁可能导致并发跑 aggregate 的其他实例 EBUSY；
3. M6.1 的 ingest 水位线机制在文件锁与 DB 锁交叉时缺安全网。

**修复建议**：所有 bin/*.mjs 的"成功路径"和"失败路径"都加 `try { ... } finally { store.close(); }`；或在 `DeliveryStore` 内部注册 `process.on('exit', () => this.close())`。

### B4. `commands/import-design.md` 内嵌 bash 中 `$URL` / `$FROM` 未做 shell 注入防护

**位置**：`commands/import-design.md:33-35,39`

**现象**：

```bash
URL=$(printf '%s' "$ARGUMENTS" | grep -oE -- '--url [^ ]+' | awk '{print $2}')
FROM=$(printf '%s' "$ARGUMENTS" | grep -oE -- '--from [a-zA-Z0-9_-]+' | awk '{print $2}')
case "$FROM" in
  figma|v0|lovable) test -n "$URL" || { echo "❌ --from $FROM 必须提供 --url"; exit 1; } ;;
```

`FROM` 经 `[a-zA-Z0-9_-]+` 白名单约束 ✓；但 `URL` 仅截到第一个空格，若用户传形如 `--url ';rm -rf foo'` 这样未引号包裹的恶意 URL，后续 Phase 3-4 中再插入到 bash 命令时（虽然现在 Phase 3 没直接插入 URL，但 frontend-agent 的实现会拿到原值并 spawn），存在 shell 注入风险。

类似模式同样出现在其他多个 commands 的 `printf '%s' "$ARGUMENTS"` 解析处。

**影响**：插件假定调用者是受信主用户，但跨会话 / relay prompt 注入场景下 ARGUMENTS 可能来自第三方 markdown，提升攻击面。

**修复建议**：(1) 用 `bin/parse-args.mjs` 集中解析（已部分存在于 bin 但 commands 未统一调用）；(2) 把所有需要嵌入 bash 子句的变量先经 `printf '%q'` 转义；(3) 至少在 URL 上加正则白名单（`^https?://[^[:space:]]+$`）。
